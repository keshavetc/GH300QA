What makes an effective prompt for GitHub Copilot?
Key characteristics of effective prompts:
  • Specificity: Clear, detailed descriptions of what you want
  • Context: Relevant information about the surrounding code
  • Examples: Sample inputs/outputs when applicable
  • Constraints: Requirements, limitations, edge cases
  • Natural language: Write like you're explaining to a colleague
Example - Bad prompt:
// function
Example - Good prompt:
// Function to validate email addresses
// Returns true if valid, false otherwise
// Should handle common cases like name@domain.com

What is the optimal level of detail in a prompt?
Balance specificity with flexibility:

Too vague:
// function that does stuff
Too prescriptive:
// Use a for loop from 0 to length-1, create empty array,
// push items if condition, return array
Just right:
// Filter array to only include items where price > 100
// Return new array, don't modify original
Rule of thumb:
  • What: Detailed (what you want to achieve)
  • How: Flexible (let Copilot choose best approach)
  • Why: Optional (helps with context)

What is the 'DRY prompting' technique?

Strategy:
  1. Create a well-documented example function first
  2. For similar functions, use minimal prompts referencing the pattern

// Second function - minimal (references pattern)

Benefit: Less repetitive commenting, consistent code patterns

What is the 'conversation' approach to prompting Copilot Chat?
Treat Copilot Chat like a pair programming session:

Effective conversation flow:
  1. Start broad: "I need to implement user authentication"
  2. Ask clarifying questions: "What's the best library for JWT in Node.js?"
  3. Request specific code: "Generate a middleware function for verifying tokens"
  4. Iterate: "Can you add error handling for expired tokens?"
  5. Refine: "Make it compatible with TypeScript"
Benefits:
  • Builds context incrementally
  • Catches misunderstandings early
  • More tailored solutions
  • Learn while coding

What is negative prompting and when is it useful?
Negative Prompting: Telling Copilot what NOT to do

Use cases:
  • Avoiding deprecated patterns
  • Preventing security anti-patterns
  • Steering away from specific libraries
Examples:
// Fetch user data WITHOUT using jQuery
// (use native fetch API)

// Hash password - do NOT use MD5 (insecure)
// Use bcrypt instead

// Validate input WITHOUT regex (for readability)
// Use simple string methods
Caution: Negative prompts are less reliable than positive ones. Better to show what you DO want.

What is progressive refinement in prompting?
Progressive Refinement: Start simple, add details iteratively

Approach:
  1. Round 1: Basic function prompt
    // Function to validate phone numbers
  2. Round 2: Add format requirements
    // Accepts (123) 456-7890 and 123-456-7890 formats
  3. Round 3: Add edge cases
    // Return false for international numbers
  4. Round 4: Add error handling
    // Handle null/undefined gracefully
Benefits: Easier to review changes, catch issues early, maintain code quality

What are anti-patterns in prompting to avoid?
Common prompting mistakes:
  • ‚ùå Too vague: // do something
  • ‚ùå Conflicting requirements: "Make it fast and use recursion"
  • ‚ùå Over-specifying implementation: "Use a while loop with counter i"
  • ‚ùå Assuming knowledge: "Use the usual pattern" (what's usual?)
  • ‚ùå No examples: Abstract descriptions without concrete cases
  • ‚ùå Ignoring context: Not keeping related files open
Instead, do this:
  • ‚úÖ Be specific about goals, flexible about implementation
  • ‚úÖ Provide examples and edge cases
  • ‚úÖ Give context through open files and comments
  • ‚úÖ Iterate and refine based on suggestions
